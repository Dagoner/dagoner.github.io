Ship O’hoi

2019.12.11
[Hugo Authors Markus L. Gundersrud Nils Oyvind Hartveit]

An interactive ship controller, when people’s hand get close to the controller (don’t have to touch), the graph which is related to the controller will pop out to help the boat driver get the information easier.

Backround Research

this is one project of tangible interaction class, the theme which we choose is ship controller

Field Observation

After visiting the USN (an university in Norway which aims to educate the captain and has the Simulated bridge), Lilaas(an company designs and produces the boat controller ), and talking with the real captain, we find that there are so many controllers on the boat and different controllers are related to different graph on the screen, besides that, there are also many screen on the boat. Well, too many graph and controllers means too much information which cause misunderstanding or mis orperation very easily.

User Research

We also talked with an captain teaching in USN who has 30 years experience of drving Ocean-going freighter, he introduced us the “boat sense” which is how the driver fell the boat(sounds like some people’s filling about their car). So, according to his introduction, the feedback of the boat controller seems not that important. But still, we don’t think only rely on sense is the best way, especially for the new driver. Moreover, in Ocean-going freighter this kind of place, it’s never too bad to make the information clearer.

My aims with Ship O’hoi

My first aim with Ship O’hoi is exploring different feedback of the boat controller and using the tangible way to make the information clearer than the traditional controller. Under that goal, we put vibration into the rudder, so the user can feel how many steps which they have rotated without looking at it. Besides that, we also use light to simulate the delay between the direction changing of engine and the rudder. It’s better to put the information in both screen and physical side for the safety reason. My other aim is simplifying the information on the bridge house, the information in the bridge house is overload because of the complexity of those big boat, and too much information is always a problem for people to use, so we try to simplify those information by putting touch sensor into the propulsion, the graph will popup or be light up when the user’s hand reaches to the propulsion, in this way, we can decrease other information’s disturbing and help user focus on what they are controlling quickly. Moreover, we also put this function into the wiper button, that’s just an example, it can be all kinds of setting button in the bridge house. In this way, we can hide all those settings on the screen and if the user wants to check them, they can just touch those physical buttons to check. In terms of changing those settings, we tried new way of changing settings, the user can choose other choices by rotating the button and confirm the choosing by just releasing the hand.

Outcome 1-Propulsion

We put the copper foil under our propulsion model and use this as the sensor, since every human being contain charge in the body, the copper can sense the change of charge by the distance between people and the copper. Accourding to that, we can just use this copper foil to make our interactive propulsion, that’s the main points of this interactive propulsion.

  • people’s hand gets close to the propulsion (a certain distance which should also be suitabel) then trigger the popup on the screen.
  • user doesn’t have to really touch the propulsion to trigger the effects because touching may cause misoperation.

Arduino Code

We use arduino library CapacitiveSensor.h to sense the hand and use Adobe XD to demonstrate the effect. The trick is controlling the Arduino Leonardo to press the keyboard and demonstrate the UI on the Adobe XD.

#include <CapacitiveSensor.h>
#include <Keyboard.h>
// 10M resistor between pins 4 & 2, pin 2 is sensor pin, add a wire and or foil if desired
CapacitiveSensor   cs_4_2 = CapacitiveSensor(4,2);        

void setup()                    
{
   // turn off autocalibrate on channel 1 - just as an example
   cs_4_2.set_CS_AutocaL_Millis(0xFFFFFFFF);    
   Keyboard.begin();
   Serial.begin(9600);
   pinMode(8,INPUT_PULLUP);
   pinMode(10,OUTPUT);
}

void loop()                    
{
    long start = millis();
    long total1 =  cs_4_2.capacitiveSensor(30);
    int button=digitalRead(8);
    
     // print sensor output 1
    Serial.println(total1);                  
    if (total1 >=350 && button==HIGH){
        Keyboard.press('1');
        digitalWrite(10,HIGH);
        delay(100);
        Keyboard.releaseAll();
    }
    if(total1 < 350 && button==HIGH){
      Keyboard.press('2');
      digitalWrite(10,LOW);
      delay(100);
      Keyboard.releaseAll();
    }
    // arbitrary delay to limit data to serial port 
    delay(10);                              
}

Outcome 2-The Wiper Button

Accroding to the technology we use above, we can also design another new interaction on the bridge—the Setting Button. For example, the wiper button which control the the boat wipe the window of the bridge. There are many this kind of setting in the bridge house which is not that important, but still leaving the button on the table, and if we put all those settings on the screen on screen that will make information too complicated. So that’s the main point of our idea.

  • the user’s hand touches the button and the setting graph will popup.
  • the user’s hand must touch the button because this kind of setting is not common operation compared with the getting close the propulsion. If we still use “getting close” to trigger the effects, that will influence the normal operation.
  • the user can choose different setting by rotate the button.
  • there are two differnt way of confirming the setting, both of them is kind right but just different way of viewing it.
  1. releasing the button to confirm the seeting, this method will reduce the action compared with click the button.
  2. cliking the button to confirm the seeting, this method is more normal and logical to express the meaning “confirm” compared the method above.

Outcome 3-The rudder

Our anthor output is the rudder. There are some bridge house using the rudder to control the direction of the boat. Since the boat is too big, the captain have to wait for the engine working after they rotate the rudder. So there is a delay between rotating the rudder and the cargo tanker really change the direction. Nowadays, some advanced ship will express this delay in screen, in this project we also put that on the physical rudder because it’s safer and better to express this important information in two different place in case of the emergency situation. So the main point of this part is

  • Indicate the change of direction in the physical rudder.
  • Show the delay that the the engine begins to work gradually.
  • give the different feedback of how much the ship will change the direction by the vibration. eg. Every 5 degree there will be a vibration so the caption have anthor feedback on vibration beyond the screen. That may be safer and give the better experience.

Arduino code

Since we use Annular RGB strip to simulate the delay, we have to use the arduino library Adafruit_NeoPixel. About the weird map in the end of the code, that’s beacues we use Rotary potentiometer to make the rudder, if we just map the analogpin to 0~14, that will be too sensitive.

#include <Adafruit_NeoPixel.h>
#define LED_PIN    10
#define LED_COUNT 13
#define BRIGHTNESS 40
Adafruit_NeoPixel strip(LED_COUNT, LED_PIN, NEO_GRB + NEO_KHZ800);

uint32_t Yellow = strip.Color(255,255,0);
uint32_t Blue = strip.Color(0,0,255);
uint32_t Black = strip.Color(0,0,0);

int analogPin= A3;
int val=6;
int val1=6;
int tail=6;
int tail2=6;
long tailInterval =1000;
long tailStart;

                    // outside leads to ground and +5V
int val_vibration = 0;  // variable to store the value read
int lastState=1;
int currentState=0;

void setup() {
  strip.setBrightness(BRIGHTNESS);
  strip.begin();
  Serial.begin(9600);
  pinMode(3,OUTPUT);
}

void loop() {
  int val2 = val1;
  val=map(analogRead(analogPin),0,1000,14,0);
  mymap();
  if(millis()>tailStart+ tailInterval){
    if(tail <val){
      tail++;
      if(val>6&& tail<6){tail=6;}
    }
    else if(tail>val){
      tail--;
      if(val<6 && tail>6){tail=6;}
      }
    else{tail=val;}
    tailStart = millis();
    //Serial.print("tail ");
    //Serial.println(tail);
    Serial.print("val ");
    Serial.print(val);
    //Serial.print(" ");
    //Serial.print("tail2 ");
    //Serial.println(tail2);
    }
    if (val1 != val2 && lastState== 1){
      vibration();
    }
    else if (val1==val2){
      digitalWrite(3,LOW);
      lastState= 1;
    }
  shownow();
}

void shownow(){
  if (val>=6){
      strip.fill(Black,0,6);
      strip.fill(Black,val+1,12-val);
      strip.fill(Yellow,6,(val-6)+1);  
      strip.fill(Blue,6,(tail+1)-6);
      
    if(val<tail){
      strip.fill(Black,tail+1,12-(tail));
      strip.setPixelColor(val, Yellow);
    }
    else if(val==tail){
      strip.fill(Blue,6,tail-6+1);
    }
  }

  if (val<6){
    strip.fill(Black,7,6);
    strip.fill(Black,0,val);
    strip.fill(Yellow,val,(6-val)+1);
    strip.fill(Blue,tail,7-tail);
    if (val>tail){
      strip.fill(Black,0,tail);
      strip.setPixelColor(val,Yellow);
    }
    else if(val==tail){
      strip.fill(Blue,tail,6-(tail-1));
    }
  }
  strip.show();
}

void vibration(){
  digitalWrite(3,LOW);
  delay(150);
  digitalWrite(3,HIGH);
  delay(150);
  lastState=-1;
}

void mymap(){
  int x=analogRead(analogPin);
  if(x>0 && x<65){val1=13;}
  if(x>75 && x<140){val1=12;}
  if(x>150 && x<215){val1=11;}
  if(x>225 && x<290){val1=10;}
  if(x>300 && x<365){val1=9;}
  if(x>375 && x<440){val1=8;}
  if(x>450 && x<515){val1=7;}
  if(x>525 && x<590){val1=6;}
  if(x>600 && x<665){val1=5;}
  if(x>675 && x<740){val1=4;}
  if(x>750 && x<815){val1=3;}
  if(x>825 && x<890){val1=2;}
  if(x>900 && x<965){val1=1;}
  if(x>975 && x<1023){val1=0;}
}

The iterations we did

In the process of designing the rudder, we make the mocking up of it and tying different level of vibration, by adjusting the vibration many times we finally choosing the right one. We also did some iterations about the light of the rudder, we tried different way of showing the step such as light dots, line. Except that, we tried different way to show what will happen if the users change the direction of the engine radically. Moreover, we did a lot of tweaks about the distance, which can trigger the popup of the graph, to make it suitable for people to probe what’s the handle for and sensitive enough to use.

My reflections on the outcome

The final outcome seems to be very useful for me, the propulsion is sensitive enough to use and it’s also very easy to make, we don’t have to make a lot of changes on the existing handle, that means it’s totally doable to produce it in the real industry. In terms of the way of popup on the screen, that may not be the most suitable way to show it, it’s more for opening a topic, I think we should make it less obvious in the real life or communicating more with the real captains to figure out which information is the more important. Moreover after observing some people using those handles, I also think how to confirm the setting still need to be discussed. Although the releasing to confirm the setting decreases the action, but I think it may be not that natural as normal people’s habit, maybe we can also use click the button to confirm the setting.
